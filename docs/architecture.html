Factory Inventory Management System

Architecture overview — a full-stack demo application with a Vue 3 SPA, a FastAPI backend, and in-memory mock data (no database).

Frontend: localhost:3000 Backend: localhost:8001 Branch: new_features Generated: July 6, 2026

System Architecture

Three tiers, one direction of dependency: the browser SPA calls the REST API over HTTP; the API serves data loaded into memory from JSON files at startup. Nothing persists — restarting the server resets all state.

Data Flow

The core loop of the app: a user changes one of the four global filters, and every view re-fetches and re-renders. Filter state lives in module-level refs inside useFilters.js, so all components share one instance.

Filter changeFilterBar.vue
→
Shared refs updateuseFilters (singleton)
→
View watch firesviews/*.vue
→
GET with query paramsapi.js (Axios)
→
Filter in memoryapply_filters / filter_by_month
→
Validate responsePydantic response_model
→
Raw data into refsallOrders, inventoryItems
→
Derive & rendercomputed properties

Convention: raw API responses go into ref()s; everything displayed (KPIs, chart series, table rows) is derived in computed() properties. Filters "all" values are omitted from query params on the client and skipped on the server. Inventory has no time dimension, so the month filter does not apply to it.

Known Gap on This Branch

Client calls endpoints the server does not implement. On new_features, client/src/api.js defines calls to /api/tasks (GET/POST/PATCH/DELETE) and POST /api/purchase-orders, but server/main.py has no matching routes. This causes the "Failed to load tasks" 404 in the browser console, and the Dashboard also references an unresolved PurchaseOrderModal component. These appear to be in-progress features.
